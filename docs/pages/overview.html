Getting started

Overview

The DGA Platforms Code Skill is directive. It tells Claude how to design and review Saudi government / semi-government interfaces so they are official, trustworthy, accessible, Arabic-RTL-correct, service-oriented, and consistent — while keeping the existing system safe and working.

Purpose

Help produce and review interfaces aligned with the DGA Platforms Code / National Design System. Aligned means: government-grade tone, accessible, RTL-correct, service-first, consistent, and engineered without breaking business logic, APIs, authentication, authorization, or validation.

⚖️

This skill aligns with DGA principles. It does not make an interface officially compliant. Official compliance requires formal review by the responsible entity. Never claim or imply official DGA compliance.

Verified Platforms Code principles

Grounded in the Platforms Code Guide v1.0:

  • Platforms Code is a national reference for designing and developing government platform interfaces.
  • It supports consistency, unified UI guidance, digital inclusion, and quality improvement.
  • It serves three beneficiary groups: designers, developers, and product managers.
  • It preserves accessibility and WCAG-aware implementation (target WCAG 2.1 AA).

When to activate

Activate when the task involves any of:

Government / semi-gov UI React · Tailwind Arabic RTL interfaces Service landing & selection Multi-step service flows Forms & inputs Upload → review → result Dashboards Admin / verification / case management Government-grade UX review

If the task is purely backend, data, or infrastructure with no UI surface, the skill does not apply.

The mandatory workflow

The skill enforces an ordered workflow before any code is edited — no skipping to edits.

#StepWhat it means
1Inspect the projectRead relevant files; identify framework, routing, styling stack, tokens, and conventions.
2Locate the seamsFind where UI ends and business logic, APIs, auth, and validation begin — and preserve them.
3Read the referencesPull only the relevant references/ guides for the task.
4Confirm scopeState what will and will not change. UI-only tasks stay UI-only.
5Plan incrementallySmallest coherent change; reuse existing components and tokens first.
6Implement, then verifyMake the change, then run available checks.
7SummarizeReport changed files and remaining manual-review items.

What must be preserved

Always safe to improve

  • How validation is presented (labels, messages, focus)
  • Visual hierarchy, spacing, and tone
  • Accessibility and RTL correctness

Never change in UI work

  • Business logic, data flow, computed results
  • API contracts (endpoints, shapes, headers, keys)
  • Authentication & authorization
  • What passes or fails validation

Prohibited behaviour

  • Copying or embedding DGA / government logos, emblems, or official marks without authorization.
  • Claiming or implying official DGA compliance, certification, or approval.
  • Inventing official rules, tokens, or values not in the official sources.
  • Changing backend, API, auth, or security logic during a UI-only task.
  • Uploading sensitive/personal data to external tools, or exposing secrets.
  • Using real personal data (names, national IDs, phones, addresses) in examples — use fake data.
  • Making a governmental interface look like a commercial SaaS / marketing product.
  • Overusing a single accent (e.g. green) until hierarchy and the true success state weaken.

Required output after each task

Every task ends with a concise report covering: Scope, Changed files, Preserved (logic/APIs/auth/validation), Checks (honestly reported), Manual-review items, and a Compliance note restating that the work aligns with DGA principles and is not official compliance.

Final quality gate

Before declaring a task done, the skill confirms:

  • Government-grade tone: calm, official, trustworthy — not SaaS / marketing styling.
  • Aligned with official sources; no invented tokens, compliance claims, or copied logos.
  • Layout and hierarchy clear; one primary action per view.
  • Arabic RTL correct, including mixed Arabic/Latin, numbers, codes, and mirrored icons.
  • Accessibility: contrast, focus, keyboard, semantics, labels, error announcements, no color-only status, touch targets, reduced motion.
  • Components and forms follow the rules; empty / loading / error states exist.
  • Content is clear, direct Arabic; messages are specific and helpful.
  • Responsive: mobile-first; no horizontal overflow; readable Arabic on small screens.
  • Engineering safety: logic / APIs / auth / validation preserved; tokens / components reused; checks run.
  • Output report delivered with changed files and manual-review items.
