Guidance

RTL & accessibility

Arabic RTL correctness and accessibility are baseline requirements, not enhancements. These are the rules that most distinguish a government-grade interface — and the ones easiest to get wrong.

Arabic RTL

Source: references/07-arabic-rtl.md.

  • Set direction at the document/layout root: dir="rtl" and lang="ar".
  • Default text alignment is right; visual flow mirrors left ↔ right.
  • Use logical CSS properties (inline-start/end, padding-inline, margin-block) instead of hardcoded left/right.
  • Mirror directional icons (back, forward, chevrons). Do not mirror logos, Latin product names, barcodes, codes, phone numbers, emails, URLs, or code.
  • Keep numbers, IDs, and Latin strings in natural LTR order even inside Arabic — isolate them so they don't reorder surrounding text.
  • Steppers, breadcrumbs, and progress flow right-to-left; "next" advances toward the left edge.

Number & code isolation

Latin codes and numbers stay LTR inside Arabic sentences using <bdi> or the .pc-num utility (direction: ltr; unicode-bidi: isolate).

Accessibility (WCAG 2.1 AA)

Source: references/08-accessibility.md.

Do

  • Meet sufficient contrast for text, icons, and interactive states.
  • Provide visible, clear focus on every interactive element.
  • Use semantic HTML: button, a, nav, main, table, label, ordered headings.
  • Label every input; announce and link errors to their field.
  • Respect reduced-motion; provide adequate touch targets.

Don't

  • Remove focus outlines without an equal-or-better replacement.
  • Convey status with color alone — pair with text and/or icon.
  • Create keyboard traps or illogical tab order.
  • Use placeholders as labels.
  • Ship tables without proper headers and scope.

Status without color alone

Every status carries a label and an icon/dot, so meaning survives for color-blind users and in grayscale.
